2026-05-04 — Quick Start: tighter fulfillment + provider-override behaviour
              · Urgent Care now auto-includes "custom pharmacy" in the
                effective pharmacy set everywhere (endpoint list, payload,
                discovery sections, recipe match) without requiring the
                user to tick the box.
              · Fulfillment question is now hidden whenever Urgent is
                selected (alone or alongside anything) — the answer is
                settled, no need to ask. Only Rx-only flows still show
                the question.
              · canContinue: Urgent never blocks step 1 anymore (custom
                is always derived). Rx-only still requires at least one
                fulfillment tick.
              · "Let the provider decide" now drops the partner-pharmacy
                discovery endpoints (partner_pharmacy_list and
                partner_pharmacy_treatment_packages) from the plan,
                hides the "Find your pharmacy IDs" and "Find your
                order_set_ids" sections, and omits pharmacy_id /
                order_set_id from the partner-pharmacy payload — the
                provider picks both at consultation, so pre-fetching them
                is dead work.
              · "Send the patient invite" section now renders TWO sample
                payloads when both partner and custom are in the
                effective set (sub-headed "For partner-pharmacy invites:"
                / "For custom-pharmacy invites:"). Applies to the
                on-screen plan, the email body, and the PDF.
              · Per-codeblock Copy buttons now know which curl they're
                copying via data-curl-idx; copyCurl() reads from a curls
                array stashed on window.

2026-05-04 — Quick Start: pharmacy fulfillment is now multi-select (and
              auto-derived when rx + urgent are both picked)
              · Step 1 "How should treatments be fulfilled?" was a forced
                single-pick (partner OR custom). Switched to multi-select
                (oc-multi) so a clinic running Rx and Urgent Care from the
                same Qualiphy account can tick both fulfillment types.
              · When the user picks BOTH Rx Treatment and Urgent Care at
                the use-case step, the fulfillment question now hides
                entirely — the answer is implicitly "both" (partner for
                Rx, custom for Urgent), so the question is just noise.
                getPharmacy() derives {partner, custom} for downstream
                consumers in that case.
              · For Rx-only or Urgent-only selections the question stays
                visible and accepts one or both.
              · canContinue updated: skip the pharmacy gate when both rx
                + urgent are selected; require at least one selection
                otherwise; keep the "Custom required for Urgent Care"
                rule when Urgent is selected without Rx.
              · pharmacy state migrated from string to Set; the HTML
                render, email body, PDF builder, and restart() all read
                the effective set via getPharmacy(). Sample /exam_invite
                payload prefers partner-pharmacy fields when both are in
                play (the primary Rx flow); the plan summary calls out
                that custom-pharmacy invites use the alternate field set.

2026-05-04 — Right-rail "Recent runs" history
              Each Try Now request (success, 4xx/5xx, or network error) is
              now logged to a Recent runs panel pinned at the bottom of the
              right-rail. Stays put as the user scrolls between endpoints —
              answers the "I just ran something, where did it go?" issue
              when the rail repopulates from a different sample on scroll.
              Stored in sessionStorage (same store as the API key — clears
              when the tab closes, doesn't leak across browser sessions).
              Capped at 20 entries, FIFO. Each row collapses to a status
              pill + endpoint + relative time; click expands to show the
              full request payload and response body. A "Reuse this
              payload" button on each row drops the request back into the
              textarea (and points the rail header at the matching
              endpoint) so you can edit and re-run without scrolling. A
              Clear all button (with confirm) wipes the list.
