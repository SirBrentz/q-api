2026-05-08 — Fix /exam_list documentation — description was wrong
              Thomas Ciccarelli flagged that the existing description
              ("Lists every exam configured on your account, including
              its ID, name, type (GFE/Rx/Urgent), and required
              questions") was wrong on multiple counts:
                * field is `title`, not `name`
                * `rx_type` is binary (1 = GFE, 2 = Rx) — there is no
                  separate "Urgent" type; urgent-care exams are just
                  rx_type:2 distinguished by their title
                * "required questions" doesn't exist as a field —
                  actual fields are `attachments_required` (count) and
                  `attachments` (string[] of attachment names)
                * `price`, `clinic_id`, `attachments_required`, and
                  `attachments` weren't documented at all
              Rewrote the endpoint description, added a Response fields
              table covering every field returned (clinic_id + per-exam
              id/title/rx_type/price/attachments_required/attachments),
              and added a Sample response showing both rx_type values,
              priced and unpriced exams, and 0/1/2-attachment cases
              including an Urgent Care exam to make the rx_type:2
              "no separate Urgent type" point concrete.

2026-05-07 — Sidebar search: results dropdown lists actual params
              Previous fix made section-level matches work (searching
              "redirect" surfaced the /exam_invite link), but Brent
              wanted to actually SEE the matching parameters
              themselves — redirect_approve / redirect_reject /
              redirect_na / redirect_missed — as distinct results, not
              just one section link. Now on page load we build a
              proper search index over (a) every sidebar link
              including its target section's full text and (b) every
              <code> name inside every .param-row, with each parameter
              becoming its own searchable item. As the user types we
              render a results dropdown below the search input grouped
              into "Parameters" and "Sections", and clicking a param
              result jumps to its row and briefly flashes it via a
              .search-flash animation so it's easy to spot. Esc clears
              the search and restores the sidebar nav.

2026-05-07 — Sidebar search: also matches docs body content
              The sidebar search only looked at each link's text + its
              data-search attribute, so common terms that only appear
              as parameter names inside the docs body — "redirect"
              (redirect_approve / redirect_reject / redirect_na /
              redirect_missed on /exam_invite), "email" (the patient
              email param), and any other field/response key — never
              surfaced. Now each sidebar link's haystack is built once
              on load to include the textContent of the section it
              points to (#ep-exam-invite, #ep-exam-webhook, etc.), so
              searching any parameter, response field, code sample
              token, or word in the docs body matches the right link.
              Cached per-link to keep filtering fast.

2026-05-06 — Sidebar: each group is now a contained card
              The five sidebar groups (Getting Started / API Reference
              / Webhook handling / Recipes / Going Live) used to read
              as one long unbroken list. Wrapped each in a soft
              container — surface bg, faint border-lt, 12px radius —
              and added a subtle divider line below each group header
              so the sub-items clearly belong to their section. All
                tokens flip per theme.

2026-05-06 — Sidebar: numbered chip uses palette tokens (theme-safe)
              The 01/02/03 chips next to the new category headers were
              shipping as solid indigo with white text — fine in light
              mode but in dark mode --indigo flips to pale lavender,
              giving white-on-lavender. Switched to the same
              --lavender-mist bg + --indigo text pattern that .s-tag
              and .s-link.active already use elsewhere, so the chip
              adapts per theme.

2026-05-06 — Sidebar: category headers now read as headers
              The "Getting Started" / "API Reference" / etc. labels in
              the left nav were rendering smaller (10px, weight 600,
              muted color) than the sub-section links beneath them
              (13px) — visually backwards. Bumped headers to display
              font, 13px, weight 800, ink color, with the numbered
              chip rendered as a filled indigo pill. Items underneath
              stay 13px weight 500, so hierarchy reads correctly.

2026-05-04 — Right-rail "Recent runs" history
              Each Try Now request (success, 4xx/5xx, or network error) is
              now logged to a Recent runs panel pinned at the bottom of the
              right-rail. Stays put as the user scrolls between endpoints —
              answers the "I just ran something, where did it go?" issue
              when the rail repopulates from a different sample on scroll.
              Stored in sessionStorage (same store as the API key — clears
              when the tab closes, doesn't leak across browser sessions).
              Capped at 20 entries, FIFO. Each row collapses to a status
              pill + endpoint + relative time; click expands to show the
              full request payload and response body. A "Reuse this
              payload" button on each row drops the request back into the
              textarea (and points the rail header at the matching
              endpoint) so you can edit and re-run without scrolling. A
              Clear all button (with confirm) wipes the list.
