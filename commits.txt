2026-05-13 — Telemetry v3 audit + robustness pass
              Brent reported that after deploying v3, some of the new

              bottom audit of every collection point in the Quick Start
              flow and tightened the page-side capture.

              PAGE-SIDE ROBUSTNESS (quick-start/index.html):
                * watchField now listens to BOTH `input` (debounced 700
                  ms) AND `blur`. Previously only blur — which doesn't
                  fire if a user pastes a value and immediately clicks
                  Continue, so qs_field_value would be missed.
                * Same `input`+`blur` doubled-up wiring for the exam-id
                  list inputs (which were also blur-only).
                * Added qs:step-flush event dispatched on every goNext
                  call. The watchField handlers listen to this and
                  flush every visible text-field value to the sheet
                  before the step advances. Final safety net: even if
                  input-debounce AND blur both miss, the step
                  transition catches it.
                * Pulled the per-field lastValByField map up one scope
                  so all flush paths (input, blur, step-flush) share
                  the same dedup state — no duplicate sends.

              APPS SCRIPT BUG FIX (dashboard/SETUP.txt):
                * Pharmacy was being stored as a SINGLE value but the
                  page allows users to pick BOTH partner AND custom
                  (it's a true multi-select with `.oc oc-multi` and
                  pickMulti, not pickSingle). Each click was
                  OVERWRITING the previous Pharmacy Choice value
                  instead of accumulating. Switched to csvAdd_ for
                  option_selected and csvRemove_ for option_deselected
                  — same pattern as Use Cases and Optional Features.

              DASHBOARD (dashboard/index.html):
                * renderPlanIntent's pharmacy block now splits the csv
                  before counting frequencies, matching the new
                  Apps Script behavior. Partner and custom each count
                  as their own data point on the chart.

              SETUP.txt:
                * Added a prominent callout about the DriveApp OAuth
                  scope. v3 uses DriveApp (for PDF upload) but v2

                  re-authorization. The script's try/catch around the
                  Drive write SILENTLY swallows auth errors, so PDF
                  Drive URL stays empty if the user redeployed without
                  granting the new scope — and there's no error in the
                  sheet to tell you why. Step-by-step instructions
                  added under PDF STORAGE.

              VERIFIED WIRED CORRECTLY (no changes needed):
                * Use Cases (gfe/rx/urgent) — option_selected /

                * Provider Override — option_selected single-choice

                  extension is non-destructive):
                    PII: Email, Company, Contact Name, Webhook URL
                         Value, Exam IDs Value, PDF Drive URL, PDF
                         Generated At
                    Per-feature flags: Opt: Prefill, Opt: Attach Docs,
                         Opt: Custom Redirect, Opt: Additional Data,
                         Opt: Skip Pharmacy Confirm, Opt: Clinic
                         Billed, Opt: Ship To Patient, Opt: Skip
                         Address Confirm
                * New events handled:
                    qs_field_value { field, value }  — actual string
                       value lands in the appropriate PII column
                       (email / company / name / webhook / examIds).
                       Values capped at 500 chars.
                    qs_pdf_generated { pdf_b64, filename }  — base64
                       decoded, Utilities.newBlob, written into a
                       Drive folder named "Qualiphy API Docs - Quick
                       Start PDFs" (auto-created on first use). File
                       URL stored in PDF Drive URL; timestamp in PDF
                       Generated At. Drive folder is owned by the
                       script-deployer, not shared publicly.
                * option_selected/deselected for optionalFeature now
                  ALSO sets per-feature Yes/No columns so you can
                  pivot in Sheets without parsing the CSV column.
                * qs_restart wipes the v3 PII columns and feature
                  flags too. (PDF Drive URL is intentionally NOT
                  wiped — the file still exists in Drive and the
                  pointer is useful history.)
                * **doGet filters PII before returning JSON.** New
                  PII_COLS list — Email, Company, Contact Name,
                  Webhook URL Value, Exam IDs Value, PDF Drive URL,
                  PDF Generated At. These are never serialized into
                  the dashboard payload, so they can't accidentally
                  render in the recent-sessions table or CSV export,
                  and they're not sniffable from browser dev tools.

              PAGE-SIDE (quick-start/index.html):
                * watchField extended to also fire qs_field_value with
                  the actual value on blur (in addition to the existing
                  qs_field_filled boolean flag). Fires for company,
                  name, email, webhook.

                  of current exam IDs on every blur (was only firing a

                * send() takes a new opts object with `large: true` to

                * downloadPdf, after doc.save, also does
                  doc.output('datauristring'), splits off the base64,
                  and posts qs_pdf_generated. Best-effort — failure
                  doesn't break the user's download.

              SETUP.txt:
                * New section E2 documents the v3 schema, the Drive
                  folder, the PII filter on doGet, and what the
                  dashboard will / won't see.
                * Section B updated to reference Code.gs (v3) instead

                * CSV export same story — column list is explicit.

2026-05-13 — Postman + OpenAPI: full polish pass for production-ready devs
              Brent reviewed the universal Postman collection and asked

              biggest UX hits. Closed both gaps on Postman, and verified
              + tightened OpenAPI in the same pass.

              POSTMAN — universal collection (postman/qualiphy-api-
              collection.json), now 40 KB up from 15 KB:
                * Auto-chaining Tests scripts on 5 endpoints capture
                  response IDs into collection variables, so devs no
                  longer copy-paste between requests:
                    /exam_list                        -> {{exam_id}}
                    /exam_invite                      -> {{patient_id}},
                                                         {{meeting_uuid}}
                    /partner_pharmacy_list            -> {{pharmacy_id}}
                    /partner_pharmacy_treatment_packages -> {{order_set_id}}
                    /custom_pharmacy_search           -> {{ncpdpid}}
                  Each script logs to the Postman console so it's clear
                  which variable just got set.
                * Saved example responses on every callable endpoint
                  (200 success + several 4xx error variants). The dev
                  sees the response shape inside Postman before they
                  even fire the request.
                * Three saved webhook examples on the reference entry
                  (event=1 Consultation Complete, event=2 Prescription
                  Confirmed with full prescription[] array, event=3
                  Prescription Tracking) so handler authors can see all
                  three payload shapes side-by-side.
                * Expanded collection variables (added pharmacy_id,
                  order_set_id, ncpdpid, meeting_uuid) with per-var
                  descriptions naming which endpoint sets them.
                * Rewritten info.description with a "Getting started in
                  30 seconds" walkthrough, auto-chain key, and the
                  field-name gotchas (patient_exam_id vs exam_id,
                  custom_pharmacy 1=Custom 2=Partner, phone_number not
                  phone, legacy exam_pos_id).
                * Environment file (qualiphy-api.postman_environment.
                  json) gained the four new vars to match.

              QUICK START — tailored generator (quick-start/index.html
              buildTailoredCollection()):
                * Caught and fixed real field-name bugs that would have
                  burned every dev who downloaded the tailored version:
                    exam_invite_id -> patient_exam_id (in resend,
                                       cancel, submit_answers, upload)
                    exam_id (singular) -> exams: [id] (array) on
                                       /exam_invite
                    partner_pharmacy_id -> pharmacy_id (on packages)
                    zip -> zip_code (on custom_pharmacy_search)
                    question_id -> no (on submit_answers)
                    file_url + filename -> file_name + file_type +
                                       file_base64 (on upload)
                    phone -> phone_number (on /exam_invite body)
                * Webhook reference body rewritten to match Qualiphy's
                  actual event=1 payload shape (was a made-up
                  approved/rejected/na string-status shape that does
                  not exist in the real API).
                * Same Tests scripts as the universal — auto-chain
                  patient_id, exam_id, pharmacy_id, order_set_id,
                  ncpdpid, meeting_uuid.
                * Sentinel-placeholder trick (S.examId etc + rawBody())
                  to work around JSON.stringify quoting integer
                  variable refs. Without this, {{exam_id}} would
                  substitute as "4226" (string) into [{{exam_id}}],

                * Generated info.description now includes a "Getting

                * Variable list expanded to include patient_id,
                  pharmacy_id, order_set_id, ncpdpid, meeting_uuid.

                * Added the int-vs-string type drift gotcha between

                * Flagged the docs contradiction on

2026-05-13 — Tracker: engaged-time semantics + dashboard scope filter
              Two related fixes after Brent flagged that "Avg Session"
              on the dashboard was wildly higher than reality, and that
              main-docs vs quick-start stats were impossible to tell
              apart.

              (1) Avg-session bug, root cause:
                  * Session ID is actually a persistent visitor ID in
                    localStorage — one row per visitor, not per visit.
                  * Duration (sec) was updated via Math.max(...) on
                    every event from page-side elapsed_sec = wall-clock
                    time since page load. A tab left open overnight
                    (or backgrounded for hours) would write an absurd
                    number into the visitor's row, and Math.max kept
                    it forever. One stuck tab poisoned the avg.

              (1) Fix:
                  * q-api/index.html + q-api/quick-start/index.html:
                    replaced the old wall-clock counter with engaged-
                    time tracking. Now we only tick the counter when
                    the tab is visible AND the user has interacted in
                    the last 30s. Hard cap of 30 min per page load so
                    even a worst-case tab can't poison the metric.
                  * dashboard/index.html: clamp legacy Duration values
                    at 30 min when mapping rows, so historical rows
                    with hour-long durations stop dragging the avg up.
                  * Renamed the stat card "Avg Session" → "Avg
                    Engaged Time" with "capped 30m" sublabel so the
                    semantics are obvious.

              (2) Docs vs Quick Start clarity:
                  * Added a Scope pill bar in the topbar:
                    [all] [main docs] [quick start]. Filters every
                    stat / funnel / breakdown below it.
                  * isQS(row) classifier: any row that landed on
                    /quick-start/ OR has plan-builder activity counts
                    as Quick Start. Otherwise it's Main Docs.
                  * Renamed "Engagement Funnel" → "API Docs ·
                    Engagement Funnel" with sub-meta "main docs ·
                    land → copy" to make it explicit which flow the
                    funnel describes.
                  * Hide the API Docs funnel panel when scope=qs
                    (it would just show near-zeros for QS rows).
                  * Dash header now shows live scope info, e.g.
                    "scope · all (42 docs · 7 QS)" so you can see the
                    split at a glance even when filter is "all".
